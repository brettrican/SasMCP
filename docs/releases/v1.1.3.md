# SassyMCP v1.1.3 — Single Source of Truth for Version

**Release date:** 2026-04-11

## TL;DR

Through v1.1.0 → v1.1.1 → v1.1.2, **the version string in the running binary never moved off `1.1.0 Beta`** because the version literal was hardcoded in 9 different places and nobody bumped them. v1.1.3 collapses all 9 down to a single source of truth in `sassymcp/__init__.py`. From now on, bumping a release is one line.

This is a meta-fix — the v1.1.2 *behavior* was correct, but the binary was misreporting itself as v1.1.0 Beta to anyone who looked at the banner.

## What was wrong

I (the AI doing the previous patches) shipped v1.1.1 and v1.1.2 without bumping any version literal. Discovery: the user launched the binary, saw `SassyMCP v1.1.0 Beta` in the banner, and asked "either way it's not the new version... the cmd executable says the wrong one." They were right. The `_security.py` / `shell.py` / `editor.py` fixes were all in the binary, but the version constant was a stale string in 9 separate files.

## Audit count reconciliation

I gave three different counts during this conversation. None matched. The actual exhaustive count:

| What I said | What it was | Truth |
|---|---|---|
| "5 places" (early) | Just `pyproject.toml` + `__init__.py` + 3 lines in `server.py` | 4 missed (`_github_client.py`, `observability.py`, `runtime_config.py`, `deploy/README.txt`) |
| "11 places" (wider grep) | Wider grep found more, but I had a regex bug — `\b1\.1\.0\b` doesn't match `v1.1.0` because `v|1` is word-char-to-word-char (no boundary) | 2 wrong: counted `setup_version` (persona schema, separate concept) and `editor.py:7` (docstring reference, not a constant) |
| Truth | 9 true SassyMCP version literals (1 source of truth + 8 consumers) | — |

The "9" is verified by running the corrected exhaustive grep `[01]\.[01]\.[0-9]+` (no leading word boundary) and excluding IP addresses, third-party dependency specifiers, and the persona schema version.

## The 9 places (now collapsed to 1)

| # | File | Old | New |
|---|---|---|---|
| 1 | `sassymcp/__init__.py:9` | `__version__ = "1.0.0"` | `__version__ = "1.1.3"` ← **single source of truth** |
| 2 | `pyproject.toml:3` | `version = "1.0.0"` | `dynamic = ["version"]` + `[tool.hatch.version] path = "sassymcp/__init__.py"` |
| 3 | `sassymcp/server.py:494` (banner) | `f"SassyMCP v1.1.0 Beta \| {tool_count} tools \| Ready"` | `f"SassyMCP v{__version__} \| {tool_count} tools \| Ready"` |
| 4 | `sassymcp/server.py:520` (argparse) | `description="SassyMCP Server v1.1.0"` | `description=f"SassyMCP Server v{__version__}"` |
| 5 | `sassymcp/server.py:564` (logger) | `f"SassyMCP v1.1.0 started \| ..."` | `f"SassyMCP v{__version__} started \| ..."` |
| 6 | `sassymcp/modules/_github_client.py:48` (User-Agent) | `"User-Agent": "SassyMCP/1.0.0"` | `"User-Agent": f"SassyMCP/{__version__}"` |
| 7 | `sassymcp/modules/observability.py:43` (`get_metrics`) | `"version": "1.0.0"` | `"version": __version__` |
| 8 | `sassymcp/modules/runtime_config.py:152` (`sassy_get_config`) | `"version": "1.0.0"` | `"version": __version__` |
| 9 | `deploy/README.txt:2` (setup guide header) | `SassyMCP v1.1.0 Beta — Setup Guide` | `SassyMCP v1.1.3 — Setup Guide` (manual text) |

After this commit, **only `sassymcp/__init__.py` carries a literal version string.** Every Python file that needs the version imports it. `pyproject.toml` reads it via hatchling's dynamic-version hook. The only file that still has to be touched manually on a release is `deploy/README.txt` because it's plain text (not Python).

## False positives I deliberately did NOT change

| File | What | Why left alone |
|---|---|---|
| `sassymcp/modules/setup_wizard.py:252` | `config["setup_version"] = "1.0.0"` | This is the **persona schema version**, not the SassyMCP version. It tracks "what schema does the persona file on disk use" so future versions can detect old setups and migrate them. It only bumps when the persona file format actually changes. Auto-bumping it on every release would break migration logic. |
| `sassymcp/modules/editor.py:7` | Docstring reference: `v1.1.2: refuses edits...` | A documentation comment about when a behavior was added, not a runtime version constant. Historical references in docstrings are intentional and stay accurate by being version-pinned. |
| `pyproject.toml:11` | `mcp>=1.0.0,<2.0.0` | Dependency on the third-party `mcp` package, not SassyMCP. |
| `pyproject.toml:15` | `pillow>=10.0.0` | Same — third-party dependency specifier. |
| `docs/releases/*.md` | Historical version mentions | Release notes intentionally reference past versions; they should never be auto-bumped. |

## How to bump a release from now on

```diff
-__version__ = "1.1.3"
+__version__ = "1.1.4"
```

That's it. One line in `sassymcp/__init__.py`. Then rebuild:

```bash
.venv/Scripts/python.exe -m PyInstaller sassymcp.spec --noconfirm --clean --distpath deploy --workpath build
```

And manually edit `deploy/README.txt` line 2 (the only non-Python file that still has a literal version).

## Note on the audit

The user explicitly asked for an exhaustive audit because I had given two different counts ("13" and "11") during this conversation. The "13" was from the v1.1.2 audit (delete-interceptor issues, not version literals — different topic). The "11" was a wider grep that double-counted and miscategorized. The honest answer is **9 true SassyMCP version literals**, all enumerated above with line numbers, all now imported from a single source. The audit process itself caught that my earlier exhaustive grep had a regex bug (`\b1\.1\.0\b` doesn't match `v1.1.0` because `v` is a word char). The corrected regex is `[01]\.[01]\.[0-9]+` with explicit IP-address filtering.
