# SassyMCP v1.1.2 — Polishing the Delete Interceptor

**Release date:** 2026-04-11

## TL;DR

Second-pass audit of `v1.1.1` turned up 13 more issues — new bypass tools the first audit missed, detection gaps (robocopy /MIR is the scary one), and three regressions in my own `is_protected_path` implementation. `v1.1.2` closes all 13 and adds **91 passing sandbox assertions** (up from 54 in v1.1.1) to lock the behavior in.

**Upgrade urgency: recommended for all users.** `v1.1.1` is still the correct floor — `v1.1.2` hardens the scope `v1.1.1` didn't touch.

---

## New bypass paths closed

### `sassy_edit_block` / `sassy_edit_multi` — unguarded file rewrite
Neither tool had any protection. An agent could call `sassy_edit_block("sassymcp/modules/_security.py", "def detect_delete_intent", "def detect_delete_intent(cmd): return False, ''\ndef _old")` and neutralize the entire interceptor without ever calling `sassy_shell` or `sassy_write_file`.

**Fix:** new `_guard_edit()` helper in `editor.py` runs `is_protected_path()` and snapshots the existing file into `_DELETE_/<name>.pre-edit.<ts><ext>` before either tool touches it. Refuses protected paths with a pointer to `sassy_selfmod_edit` for legitimate in-tree edits.

### `sassy_copy` silent overwrite
`shutil.copy2(src, dst)` overwrites an existing `dst` with no warning. `v1.1.1` fixed this for `sassy_move` but missed `sassy_copy`.

**Fix:** `sassy_copy` now refuses existing destinations (same shape as `sassy_move`) and runs protected-path checks on both ends.

### `sassy_adb_shell` — no destructive check
Gated on `validate_command` (blocklist only). `sassy_adb_shell("rm /sdcard/Pictures/*")` would execute on any connected Android device.

**Fix:** new `allow_destructive: bool = False` parameter. Runs `detect_delete_intent` by default; override required for legitimate workflows like `pm clear <pkg>`.

### `sassy_selfmod_rollback` — zero-friction destructive checkout
`git checkout HEAD -- <path>` silently discards every uncommitted change in the file. No confirmation. `v1.1.1`'s audit flagged discarding uncommitted changes as an action that should require confirmation; the tool didn't.

**Fix:** new `confirm: str = ""` parameter. Refuses without `confirm="YES"` and points the caller at `sassy_selfmod_status` to review pending changes first.

## Detection gaps closed

### The robocopy /MIR footgun
`robocopy V:\empty V:\Projects\SassyMCP /MIR` mirrors empty into SassyMCP and **deletes every file in the destination that isn't in the source**. One call wipes the tree. `v1.1.1`'s keyword set didn't include `robocopy`.

**Fix:** two new destructive patterns — `robocopy ... /MIR` and `robocopy ... /PURGE`. Plain `robocopy` without those flags is still allowed (it's the normal copy mode).

### Other new destructive patterns

| Pattern | Example | Why it's destructive |
|---|---|---|
| `Out-File -Force` | `Get-Content foo \| Out-File -Force bar` | Overwrites target file |
| `Out-File -Overwrite` | same | same |
| `New-Item -Force` | `New-Item -Force existing.txt` | Replaces an existing file |
| `copy /y` | `copy /y foo bar` | CMD silent overwrite |
| `xcopy /y` | `xcopy /y /e src dst` | CMD silent overwrite |
| mid-command `>` | `type foo > bar.txt` | Truncate-by-redirect that `v1.1.1` only caught at start-of-segment |

### PowerShell `-EncodedCommand` base64 payloads
`powershell -EncodedCommand ZABlAGwAIABmAG8AbwA=` is base64-encoded UTF-16-LE for `del foo`. `v1.1.1` saw the `-EncodedCommand` flag but scanned the literal base64 string (which contains no delete keywords) and passed it through.

**Fix:** new `_decode_powershell_base64()` helper. When the scanner sees `-EncodedCommand` / `-Enc` / `-E` inside a PowerShell wrapper, it base64-decodes the payload as UTF-16-LE and recursively scans the decoded text. Undecodable payloads pass through (rely on other defenses).

**Note:** this required restructuring `_scan_segment` to accept both lowered and original-case segments — base64 is case-sensitive.

### Assignment prefix strip — `$null = ri foo`
`v1.1.1`'s first-word check saw `words[0] == "$null"` and missed the `ri foo` that followed.

**Fix:** new `_PS_ASSIGNMENT_PREFIX` regex strips `$var =` / `$null =` before first-word matching.

### `Set-Content -Value` false positive (regression from v1.1.1)
`v1.1.1`'s `Set-Content -Value ''` pattern was too loose:
```python
r"\bset-content\b[^|;&\n]*-value\s+['\"]?\s*['\"]?"
```
This matched any `Set-Content ... -Value` followed by any characters — even `Set-Content -Path foo -Value "hello world"` got flagged as destructive.

**Fix:** tightened regex requires a literal empty quoted string:
```python
r"\bset-content\b[^|;&\n]*?-value\s+(?:''|\"\")(?:\s|$)"
```
Legitimate `Set-Content` calls now pass through.

## Regressions in v1.1.1 `is_protected_path`

I used `Path(path).absolute()` in `is_protected_path()` instead of `.resolve()` — deliberate, to preserve symlinks in the interceptor's **move** logic. But `absolute()` doesn't collapse `..` or follow symlinks for the **protection check**, and three bypasses dropped out:

| # | Path | Before v1.1.2 | After v1.1.2 |
|---|---|---|---|
| R1 | `V:\Projects\SassyMCP\_DELETE_\..\sassymcp\modules\shell.py` | **not protected** (the `"_DELETE_" in parts` exemption fired before the `..` was resolved) | protected |
| R2 | `V:\Projects\SassyMCP\..\SassyMCP\sassymcp\modules\shell.py` | **not protected** (unnormalized parts never match the protected-root prefix) | protected |
| R3 | Windows 8.3 short names (e.g. real `PROGRA~1\`) | theoretically bypassable | `resolve()` uses `GetFinalPathNameByHandle` to expand |

**Fix:** `is_protected_path()` now uses `resolve(strict=False)` for the check (and for each protected-root comparison). The interceptor's **move** logic still uses `absolute()` — those are two different concerns:

- **Protection check** — "does this target something inside a protected tree?" Use the real path. Use `resolve()`.
- **Move operation** — "move this literal directory entry" (so symlinks stay symlinks). Use `absolute()`.

Both behaviors are correct, for different reasons. The `v1.1.1` bug was using `absolute()` for both.

**Self-inflicted wound in the RED phase:** during this commit's TDD cycle, my failing test for `sassy_edit_block`'s protected-path refusal **actually rewrote `sassymcp/modules/shell.py`** — the tool had no protection, the test ran the edit, and my v1.1.1 `shell.py` fix was temporarily nuked until I restored it from HEAD. After the fix landed, I re-wrote the test to target paths the tool refuses on protection grounds rather than trusting it to refuse on a no-match path.

## Upgrade notes

- **Source-based installs:** pull, restart your MCP client.
- **`deploy/sassymcp.exe` users:** rebuilt. New SHA256 in the commit. Restart the client after dropping in the new binary.
- **Breaking (light):** `sassy_adb_shell` destructive commands now require `allow_destructive=True`. If you had workflows running `rm` through it, add the flag.
- **Breaking (light):** `sassy_selfmod_rollback` requires `confirm='YES'`. If you scripted rollbacks, add the argument.
- **Breaking (light):** `sassy_copy` now refuses existing destinations. If you were relying on silent overwrite, `sassy_safe_delete` the destination first.
- **Breaking (light):** `sassy_edit_block` / `sassy_edit_multi` now snapshot existing files into `_DELETE_/` before each edit. Your `_DELETE_/` folder will accumulate pre-edit snapshots; clean them with `sassy_safe_delete` on the `_DELETE_` subfolder when you're confident in the new state.
