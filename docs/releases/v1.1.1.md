# SassyMCP v1.1.1 — Delete Interceptor Hardening

**Release date:** 2026-04-10

## TL;DR

The `v1.1.0` delete interceptor guarded exactly one tool (`sassy_shell`) and had seven bypass paths plus several parser holes. `v1.1.1` closes every known bypass, gates every destructive tool through a single detection layer, and writes a forensic audit record for every interception.

**Upgrade urgency: recommended for all users.** If your agent runs in an unattended loop, upgrade now.

---

## What was broken

1. **`sassy_session_send` / `sassy_session_start(command=)`** — persistent terminals bypassed the interceptor entirely. Any `del foo` typed into a running PowerShell session was executed without validation.
2. **`sassy_linux_exec`** — remote SSH commands had no delete check.
3. **`sassy_write_file(mode="rewrite")`** — silently clobbered existing files. No backup, no staging.
4. **`sassy_move`** — silently overwrote existing destinations.
5. **`sassy_selfmod_edit` / `sassy_selfmod_write`** — on a syntax-check failure for a newly-created file, the file was unlinked. A crafted bad-syntax submission could weaponise this into arbitrary file deletion.
6. **`sassy_audit_clear`** — `unlink`'d the audit log and its rotations with no confirmation. First move of any attacker was to wipe the forensic trail.
7. **PowerShell alias `ri`** — not in the keyword set. `gci *.tmp | ri` passed straight through.
8. **Shell wrappers** — `powershell -c "del foo"`, `cmd /c del foo`, `bash -c "rm foo"`, `wsl -- rm foo` all slipped through because detection only looked at the outer command name.
9. **`.NET` direct calls** — `[System.IO.File]::Delete(...)` wasn't keyword-matched.
10. **`Clear-Content`, `Set-Content -Value ''`, truncate-by-redirect (`> file`)** — not classified as destructive.
11. **Windows path parser** — `shlex.split(command, posix=True)` mangled backslashes: `del C:\Users\foo` became `["del", "C:Usersfoo"]`. The interceptor then reported "blocked and moved" while the actual target was untouched.
12. **`Path.resolve()` on symlinks** — deleting a symlink dragged the real target into `_DELETE_/` instead of staging the symlink.
13. **No intercept audit records** — when the interceptor fired, nothing was logged. If the caller discarded the response, forensics were lost.
14. **Interceptor could delete the SassyMCP source tree** — no protected-path list.
15. **`_DELETE_` recursion** — running `del _DELETE_` to clean the staging folder tried to move `_DELETE_` into itself.

## What changed

### Detection (`_security.py`)
- `detect_delete_intent` now catches PowerShell aliases (`ri`, `rni`), Sysinternals (`sdelete`, `sdelete64`), shell wrappers (`powershell/pwsh/cmd/wsl/bash/sh/zsh -c`), `.NET File.Delete`/`Directory.Delete`, `Clear-Content`, `Set-Content -Value ''`, truncate-by-redirect, and `Move-Item $null`.
- Wrapper payloads are **recursively scanned** so nested delete keywords are caught regardless of depth.
- New `is_protected_path()` helper marks the SassyMCP source tree, `~/.sassymcp/`, and any `_DELETE_/` folder as refused-by-default. Every guarded tool checks it.

### Shell parser (`shell.py`)
- Windows-safe splitting: `posix=False` on Windows preserves backslashed paths.
- `absolute()` replaces `resolve()` so symlinks are moved as symlinks.
- CMD flag allowlist instead of "any short `/X` token is a flag" — no more losing POSIX paths like `/tmp/foo`.
- Protected-path refusal for every target.
- Staging-folder `mkdir` wrapped in error handling.
- Raw command piped into the audit helper.

### Bypass closure
- **`sassy_session_send` and `sassy_session_start(command=)`** now gate through `validate_command` + `detect_delete_intent`.
- **`sassy_linux_exec`** refuses destructive commands on the remote host.
- **`sassy_write_file(rewrite)`** snapshots the existing file into `_DELETE_/<stem>.overwrite.<ts><ext>` before overwriting, and refuses protected paths.
- **`sassy_move`** refuses any destination that already exists (no silent clobber) and refuses protected src/dst.
- **`sassy_safe_delete`** uses `absolute()` and refuses protected paths.
- **`sassy_selfmod_*`** on syntax-check failure for a new file, renames to `<name>.bad.<ts>` instead of `unlink`.

### Forensics (`audit.py`)
- New `log_intercept()` writes a record for every interception — keyword, raw command, targets, results — to both `audit.log` and `audit.jsonl`.
- `sassy_audit_clear` now requires `confirm='YES'` and **rotates** the log to `audit.cleared.<ts>.log`. The log is never unlinked.

Run with: `V:\tools\python\python.exe _test_intercept.py`

## Upgrade notes

- **Source-based installs:** pull, restart your MCP client. No config changes required.
- **`deploy/sassymcp.exe` users:** rebuild the binary from this tree before shipping — the previous release binary contains the unfixed interceptor.
- **Breaking:** `sassy_audit_clear` now requires `confirm='YES'`. Callers that relied on zero-arg clearing will see a refusal message until updated.
- **Breaking:** `sassy_move` now refuses existing destinations. Callers relying on silent overwrite must `sassy_safe_delete` the destination first.
- **Breaking (light):** `sassy_write_file(mode="rewrite")` on an existing file now creates a `_DELETE_/<stem>.overwrite.<ts><ext>` snapshot. If you were relying on the filesystem staying clean, be aware.
