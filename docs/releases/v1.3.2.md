## v1.3.2 — SSRF hardening + onboarding docs

### What changed

A pre-release pass through the audit pipeline (Mercury-2) flagged a real
SSRF in the URL validator and surfaced that the README never explained
the actual onboarding flow. Both are fixed in this release.

- **SSRF: DNS-resolved private IPs are now blocked.** `validate_url`
  previously only checked literal IP addresses and a tiny hardcoded
  hostname allowlist, so `evil.example.com` with an A-record pointing at
  `10.0.0.1` would slip through. The validator now resolves hostnames
  via `getaddrinfo` and rejects every returned address that falls in
  the loopback / link-local / RFC1918 / cloud-metadata ranges. The
  `allow_private=True` opt-in (used by `crosslink` for LAN traffic) is
  unchanged. Also added: a 2048-character URL length cap, broader
  metadata-name allowlist (`metadata.azure.com` alongside the GCP one),
  and fail-closed behavior on `gaierror`.
- **README onboarding section rewritten.** The old "Guided Setup" block
  was a 6-row table that never explained the actual procedure. It now
  documents:
  - The recommended onboarding order (license → persona → GitHub → SSH
    → optional tools → status check → auth tokens) with the actual tool
    invocations and what each writes to disk.
  - The full 11-field persona questionnaire — every `sassy_setup_wizard`
    parameter with valid values and defaults — pulled directly from the
    tool signature so it stays accurate as the wizard evolves.
  - The trigger phrases the onboarding hook fires on, so users know
    how to summon the flow from any MCP client.
- **Mercury audit tooling committed** under `tools/mercury_audit_sassymcp.py`
  and `audits/`. The script splits the source into three chunks
  (core+oauth+config / modules-A / modules-B), POSTs each to
  `api.inceptionlabs.ai`, and writes per-chunk JSON+prose reports.
  Source bundles (`audits/*_bundle.txt`) are gitignored — they're
  full-source dumps that get regenerated on every run.

### Other findings deferred

The audit also surfaced 20+ high-severity items (ADB path validation,
registry-write whitelist, crosslink default-bind, audit-log secret
redaction, web_inspector skipping `validate_url`, etc.) and two license
findings (`get_allowed_groups` returning all groups in beta-mode,
`_SIGNING_SECRET` shipping a placeholder default). The license findings
are intentional per the in-code comment that defers tier gating until
the keying system is finalised. The high-severity queue is tracked for
v1.4.x.

### What still works exactly the same

- All distribution surfaces from v1.3.1 — portable zip, standalone exe,
  license-gated one-line PowerShell, Cloudflare Tunnel quick-start.
- All updater tools, shell-interceptor hardening, and safe-delete
  behavior are unchanged.
