## v1.2.0 — Shell Interceptor Hardening

### What broke

The `sassy_shell` interceptor uses two parallel detection paths:

1. **Keyword detection** — first token of the command is in
   `_DELETE_KEYWORDS` (`rm`, `del`, `remove-item`, `unlink`, etc.).
2. **Pattern detection** — regex matches against the full command for
destructive constructs that aren't bare keywords (`new-item -force`,
`out-file -force`, `truncate-by-redirect`, etc.).

Both paths shared a single response handler: parse the command,
treat every non-flag token as a deletion target, and move each
target into a `_DELETE_/` staging folder in its parent directory.

This is correct for keyword matches — `rm foo bar baz` really does
intend to delete `foo`, `bar`, and `baz`. It is **wrong** for pattern
matches, where the dangerous construct is one piece of a larger command
and the target cannot be identified by tokenization alone. A
PowerShell command like:

```powershell
Start-Process -FilePath "C:\Users\Admin\.local\bin\uv.exe" `
  -ArgumentList @("--directory","V:\Projects\SassyMCP","run","python", ...) `
  -RedirectStandardOutput "V:\Projects\SassyMCP\logs\bridge.out.log"
```

would trip the `truncate-by-redirect` pattern (because of the `>` in
`-RedirectStandardOutput`'s implementation) and the response handler
would walk the tokenized command, find `uv.exe` and `python.exe` as
real on-disk paths, and "safe-delete" them into staging. Service
breakage followed until the binaries were manually moved back.

### What was fixed

`sassy_shell.run()` now branches on detection type. Keyword matches
retain the existing behavior. Pattern matches block the command and
return an explanatory message without invoking the staging mover at
all.

```python
# Before
is_delete, keyword = detect_delete_intent(command)
if is_delete:
    targets = _parse_delete_targets(command)
    return await _safe_move_to_staging(targets, keyword, command)

# After
is_delete, keyword = detect_delete_intent(command)
if is_delete:
    kw_root = keyword.split(":", 1)[-1]  # strip "encodedcommand:" prefix
    if kw_root in _DELETE_KEYWORDS:
        targets = _parse_delete_targets(command)
        return await _safe_move_to_staging(targets, keyword, command)
    return (
        f"Command blocked (safety): matched destructive pattern "
        f"'{keyword}'. No files were moved.\n"
        f"Use sassy_safe_delete() for intentional deletions."
    )
```

### Detection coverage is unchanged

This release does **not** weaken any detection. Every command that
was blocked before is still blocked. The change is purely in the
response: pattern matches now refuse without acting on the filesystem.

### What you should do

- **Check `_DELETE_/` folders** in your project tree, `.venv/Scripts/`,
  `~/.local/bin/`, and anywhere else `sassy_shell` may have run
  recently. Items found there from prior false positives can be
  safely restored.
- **Rebuild the frozen binary** if you ship `dist/sassymcp.exe` —
  the source-only fix won't help users who run the compiled build.
- **No config changes required.**

### Recommended follow-ups (not in this release)

- Pre-strip quoted strings before pattern matching, so `>` characters
  inside string literals don't trigger the redirect rule.
- Add an `--allow-pattern` opt-in flag for power users who explicitly
  want to bypass a single matched pattern.
- Surface false positives via `sassy_audit_search` so users can spot
  noisy patterns before they bite.
