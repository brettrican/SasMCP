## v1.3.0 — Kali-style In-Server Version Updater (DRAFT)

**Status:** in development. `__version__` is `1.3.0-dev`. Do not tag.

### What's new

The new `updater` tool group adds four MCP tools that mirror Kali's
`apt update` / `apt list --upgradable` / `apt show` / `apt upgrade` flow,
operating against this repo's GitHub Releases (or the gated
`sassyconsultingllc.com` endpoint when a license key is set).

| Tool | apt analogue | Returns |
|------|--------------|---------|
| `sassy_update_check` | `apt update` | current vs. latest tag, upgradable bool, release URL |
| `sassy_update_list` | `apt list --upgradable` | per-asset rows (name, size, download URL, content type, download count) |
| `sassy_update_changelog` | `apt show <pkg>` | release notes body for any tag |
| `sassy_update_apply` | `apt upgrade` (download phase only) | downloads the chosen asset to `%LOCALAPPDATA%\SassyMCP\updates\` and prints the run command |

### Design notes

- **Apply does not execute the installer.** MSIs need UAC. The user runs
  the returned `msiexec /i ...` command when they're ready. This matches
  apt's contract: download is automatic, state-changing install is a
  deliberate command.
- **License-aware download path.** When `SASSYMCP_LICENSE_KEY` is set in
  the environment, `sassy_update_apply` rewrites the URL to
  `https://sassyconsultingllc.com/download/sassymcp/windows/<asset>?key=...`
  instead of the GitHub Releases URL.
- **5-minute cache** on `_refresh()` to avoid hammering the GitHub API
  on every check call. `force=True` bypasses it.
- **Version comparator** parses `1.3.0-dev`, `v1.2.0`, `V1.2.0`,
  `1.2.0-rc1` — pre-release suffixes sort BELOW the matching release
  (so `1.3.0-dev < 1.3.0`).

### Shell interceptor follow-ups (carried over from v1.2.0)

All three "Recommended follow-ups" from the v1.2.0 release notes shipped:

- [x] **Pre-strip quoted strings before pattern matching.** A literal `>`
      inside `"V:\\logs\\bridge.out.log"` can no longer trigger the
      `truncate-by-redirect` rule. `_strip_quoted_strings()` neutralizes
      single/double/backtick-quoted runs (replacing contents with `x`)
      before the destructive-pattern regexes run. Keyword/wrapper matching
      is unaffected — only pattern matching uses the stripped form.
- [x] **`allow_pattern` opt-in flag on `sassy_shell`.** Power users who
      know a specific pattern is a false positive can pass
      `allow_pattern="truncate-by-redirect"` (or `allow_pattern="*"`) to
      bypass that single match for one call. **Keyword matches
      (rm/del/Remove-Item/etc.) are NEVER bypassable** — only regex
      patterns. Every bypass is recorded as a `pattern_bypass` audit
      entry for forensic review.
- [x] **Pattern hits surface via audit.** Two new audit events —
      `pattern_block` (refused) and `pattern_bypass` (allow_pattern
      used) — are written to `~/.sassymcp/audit.{log,jsonl}`. New tool
      `sassy_audit_false_positives(count, include_bypasses)` returns
      them as a one-row-per-event table for triage.
